# Головоломка — базовые правила для Claude Code

Этот файл загружается автоматически в каждой сессии. Модульные правила лежат во вложенных `main/<app>/CLAUDE.md` и подхватываются при работе с файлами соответствующего приложения.

Связанные документы:
- [docs/PROJECT.md](docs/PROJECT.md) — стек, структура, точки входа, соглашения
- [docs/INDEX.md](docs/INDEX.md) — карта документации

## Язык

- Отвечай по-русски. Документация, коммит-сообщения, ТЗ — по-русски, если явно не сказано иначе.

## Приоритеты и ядро поведения

- Явные инструкции пользователя в текущем чате > этот файл > вложенные `CLAUDE.md`. При конфликте и недостатке контекста — спроси.
- Кратко: суть → шаги → результат. Без вступлений и пересказов уже сказанного.
- Экономь контекст: не дублируй большие блоки кода в ответ без запроса; ссылайся на пути (`main/home/urls.py:42`), давай короткие выдержки, а не файлы целиком.
- Для широкого поиска по репозиторию и параллельных независимых подзадач — используй субагента `Explore` (не забивай основной контекст).
- Крупные задачи — разбивай на подзадачи с порядком и зависимостями. Не смешивай несколько больших целей без явного запроса.
- Если чат длинный и контекст перегружен — предложи чекпойнт в `tz/STATUS.md` и новый чат.

## Старт новой сессии

1. Прочитай `docs/PROJECT.md` (стек, структура).
2. Если есть `tz/STATUS.md` — прочитай, чтобы продолжить с текущего этапа. Коротко сообщи пользователю, с какой точки подхватываешь.
3. Если `docs/PROJECT.md` отсутствует — запусти `/bootstrap` для первичной настройки.

## Тип проекта: Legacy

Код уже написан. Действуй по legacy-режиму:

- **Минимальные правки**: меняй только то, что попросили. Не «улучшай» окружающий код, не переименовывай переменные, не реструктурируй файлы без запроса.
- **Уважай существующие паттерны**: если в коде используются старые подходы — следуй им ради консистентности. Предложения по улучшению — отдельно, не молча переписывай.
- **Зависимости**: не бампай версии, не заменяй библиотеки, не правь `requirements.txt`/`package.json`/`yarn.lock` без явного запроса. Не выдумывай версии — спрашивай менеджер пакетов.
- **Незнакомая зона**: перед правкой — прочитай соседний модуль (импорты, вызовы, тесты). Не угадывай поведение.
- **Тесты**: если есть — прогоняй до и после изменений, сообщай результат.

## Качество кода

- Чисто: ни закомментированного кода, ни неиспользуемых импортов, ни мёртвых переменных. Убирай при правках.
- Имена — по смыслу, в стиле проекта (`get_user_orders`, а не `data2`).
- Комментарии только там, где неочевидная логика, компромисс или контракт. Не пиши `# увеличиваем счётчик`.
- DRY: блок, повторённый ≥3 раз — выноси в функцию/модуль, если это не вредит читаемости.

## Безопасность

- Секреты (ключи, токены, пароли) — только через переменные окружения / `local_settings.py`. Никогда не хардкодь.
- `.env`: если нужен — создавай `.env.example` с плейсхолдерами, реальный `.env` не коммить.
- Деструктивные операции (`rm -rf`, `DROP`, `TRUNCATE`, `git reset --hard`, `--force`, `--no-verify`) — только после явного подтверждения пользователя.

## Git

- Формат коммитов: `type(scope): описание` (conventional commits). Типы: `feat`, `fix`, `refactor`, `docs`, `test`, `chore`. Описание — на русском.
- Без явной просьбы — не коммить, не пушить, не создавать ветки.
- Никогда без явного запроса: `--force`, `--no-verify`, `reset --hard`, `rebase -i`, force push в `main`/`master`.
- Перед коммитом — `git status`; не стаж `.env`, ключи, токены. Заметил секрет в staged — предупреди.

## Самокоррекция при ошибках

- Ошибка команды/сборки/линтера — сначала **читай вывод**, диагностируй, потом чини. Не ретраить слепо.
- Лимит: **3 попытки** на одну и ту же ошибку. После — сообщи диагноз и варианты.
- Если правка сломала сборку и быстрый фикс не очевиден — откатывай и сообщай.

## Документация (`docs/`)

- Хранить в `docs/`. Крупные темы — отдельные файлы в тематических подпапках (`docs/architecture/`, `docs/api/`, `docs/deployment/`).
- Без монолитов: разбивай и кросс-линкуй.
- Карта — `docs/INDEX.md`. Новый файл в `docs/` — обнови карту.
- Если создан `docs/skills-index.md` — добавь ссылку в `INDEX.md`.

## Карта кода и правила

- После изменений, затрагивающих **пути, новые файлы/папки или границы модулей**, уже отражённые в `docs/PROJECT.md`, `docs/INDEX.md` или модульных `CLAUDE.md` — сразу обнови их. Карта и правила не должны врать о репозитории.
- После новой фичи или серьёзного рефакторинга в стабильной важной зоне — обнови/добавь модульный `CLAUDE.md` (ключевые пути, ответственность, контракты).
- При переименовании/перемещении файла — сразу правь карту и соответствующие `CLAUDE.md`.

## Workflow ТЗ (`tz/`)

- Перед реализацией — единое ТЗ в `tz/TZ_<тема>.md` (имя транслитом, латиница, подчёркивания).
- До **одобрения** ТЗ — никакой массовой реализации; сам `TZ_*.md` править можно.
- После одобрения — задачи из пунктов ТЗ: `TASK_<short>.md` (транслит, уникальный на ТЗ); можно класть в `tz/tasks/` и линковать из `TZ_*.md` и `tz/STATUS.md`.
- В каждой задаче — чек-лист прогресса (`- [ ]` / `- [x]`).
- Многоэтапные работы — веди `tz/STATUS.md` (или `TZ_<тема>_STATUS.md`): текущий этап, последний выполненный шаг, ссылки на `TZ_*.md`, `TASK_*.md`.

## Skills (`.claude/skills/`)

- Claude Code сам загрузит skill, если он есть в `.claude/skills/<тема>/SKILL.md`. Перед выводами о стеке/тулчейне — проверь, есть ли подходящий skill. Следуй ему, если не противоречит ТЗ или явным инструкциям.
- Skills **не создавай** без явного подтверждения пользователя («да», «создай скилл», «добавь»). Без подтверждения — только текст/план в чате.
- После подтверждения: `.claude/skills/<тема>/SKILL.md`; индекс — `docs/skills-index.md` (тема, путь, «когда читать»). При первом создании — положи ссылку в `docs/INDEX.md`.

## MCP и инструменты

- Не вызывай MCP-инструменты без ясной пользы для текущего шага. Не предполагай, что сервер включён, если пользователь его отключил.
- Предпочитай встроенные инструменты (`Read`, `Edit`, `Write`, `Glob`, `Grep`) перед `Bash` там, где они применимы.
